Sequences
& Recurrence

How recurrent networks read logs and event streams one line at a time, carry a memory, and how the LSTM keeps that memory alive across a long session.

What this session covers

  2. The recurrent neuron, and reading with a memory
  3. Why that memory fades, and how the LSTM holds on
  4. Variants and shapes, and where recurrence runs in security

What makes data a sequence

A sequence is an ordered list of items, where the position of each item is itself part of the information. Order is not decoration; it is data.

a bag
Order does not matter

a shopping list: milk, eggs, bread. Shuffle it and the groceries come out the same. Only the contents count.

a sequence
Order is the message

a recipe: crack, whisk, then heat. Shuffle those steps and breakfast fails. The same steps in a new order mean something else.

Same items, different order, different outcome. A model that only counts what appeared sees the bag; a sequence model reads the order.

So much data is already a sequence

Once order counts as information, a wide range of security data reveals itself as a sequence, the same shape beneath different surfaces.

Text and email
words, one after another.
A login session
commands, in the order they run.
Network traffic
packets, in the order they cross the wire.
System logs
events, in time order.
An executable
bytes, read front to back.

Different surfaces, one underlying shape: items, in order. That is exactly what the models in this session are built to read.

An attack unfolds as an ordered sequence

A system log is not a bag of events. It is a sequence in time, and the order carries the meaning.

08:01:07failed password for admin (x18)
08:14:22accepted password for admin
08:15:03sudo: admin opened root shell
08:22:41scp 2.1 GB to 185.62.40.9

Each line alone is ordinary. Admins mistype passwords, sign in, escalate, and copy files every day. Read in order, they trace a clear progression: brute force a login, break in, escalate to root, and exfiltrate the data.

A model that only counts which log lines appeared would miss this entirely. The task requires a model that reads the order.

Part 01 of 03

Reading in order

Why a sequence needs a memory, not just a tally.

Same events, different meaning

Order is not a detail; it reverses the meaning. The same events in a different sequence can read as routine or as an alarm.

08:00login
08:02read file
08:05logout

an ordinary day at work.

08:00read file
08:02login
08:05logout

reading a file before a login should not be possible.

Identical events, opposite meaning. The order is the signal.

Think it through

Sequence, or not?

Order carries the signal here, but it does not carry it everywhere. For each task, decide: does the meaning live in the order of events, or would a model that just counts features do just as well?

is this one email phishing? did this account get taken over? is this file malware? is this a fraud ring?

Where it bites. Two of these hinge on order, a login-and-action history and a chain of transactions; two are often fine as a bag of features. Reaching for an RNN when order does not carry the signal just adds cost and fragility.

You already use one every day

A phone keyboard reads what has been typed so far, in order, and suggests the next word. That is a sequence model with a memory, and it is already familiar.

airport
office
gate
See you at the |

reads the words so far, then predicts the next

Nothing here is exotic. The model reads in order, holds a running sense of the sentence, and guesses what follows.

Swap the sentence for a log or a session, and predict the next word becomes predict the next event. When the real next event is not the expected one, that surprise is the signal a defender wants.

Reading with a memory

A reader takes in a sentence one word at a time, holding a running sense of what came before. A sequence model operates the same way: it keeps a small summary and updates it at every step.

That running summary is the hidden state, the model's memory of the sequence so far, carried forward and updated with each new event.

When the model reads the 2 GB copy, the memory already encodes that this account just failed a burst of logins and then signed in successfully. Context accumulates.

the loop, in words

one event + the memory so far
→ an updated memory

\( h_t = f(h_{t-1},\, x_t) \)

The new state combines the old state with the new input. Repeated across the whole sequence.

Part 02 of 03

Recurrent networks

A loop that carries memory forward, and the moment it forgets.

A loop that reuses one cell

Recurrent means exactly this: a single small cell with a loop. Its memory feeds back as an input to the next step, so the same cell, with the same weights, runs at every step.

one cell, looping

unrolled across the sequence
c
→
c
→
c

the same cell, drawn once per log line

The key point: one small network, one set of weights, reused at every step, not a fresh network for each line. That reuse is what lets a single model read a sequence of any length.

Inside the cell is a familiar neuron

The update written earlier as \( h_t = f(h_{t-1},\, x_t) \) hides nothing new. Open up \( f \) and it is the same neuron used across neural networks: each input times its weight, summed, plus a bias, through an activation.

in a feedforward or convolutional network
\[ h = \phi\!\left( w_1 x_1 + w_2 x_2 + \cdots + b \right) \]

one bundle of inputs: features times their weights, a bias, then an activation.

in a recurrent network
\[ h_t = \phi\!\left( W_x\, x_t + \textcolor{#3b56d8}{W_h\, h_{t-1}} + b \right) \]

the very same form, with a second bundle: the memory from the previous step, weighted by its own \( W_h \).

The same equation you have met before, features × weights + bias, through an activation. The one new idea is that a recurrent neuron also takes its own previous output as an input.

The recurrent network unrolled

A recurrent network is that loop drawn out. The same small network repeats at each step, passing its memory along the chain.

from the paper

Elman (1990)[1] set the template: "One approach is to represent time implicitly by its effects on processing rather than explicitly," letting a network's own memory carry the sequence.

Think it through

Predict the next event

The model has read three log lines so far: a burst of failed logins, then an accepted login, then a sudo to root. Two questions: what would an ordinary administrator session do next, and which next line should the model find alarming?

open a config file restart a service log out install an update copy 2 GB to an outside host

Consider the adversary. Each option looks ordinary on its own. The model's only edge is the three lines it already read, the memory of how this session began.

What the memory actually holds

The hidden state is a running summary. Read as plain English, here is the story it builds as each log line arrives.

08:01failed logins
→
someone is guessing passwords
08:14login: admin
→
and now they are in
08:15sudo: root
→
with full administrator control
08:222 GB copied out
→
and data is leaving, this is an attack

Each line updates the summary, and the context carries forward. By the last line, the memory has turned four ordinary events into one clear story, something a simple tally of events could never do.

The memory fades over long sequences

As the chain grows long, the early steps wash out. By the end of a long session, the network has all but forgotten how it began.

The effect is analogous to the game of telephone: a message whispered down a long line arrives garbled, and the first words are lost.

This limitation is decisive here. The successful login that opens the intrusion can sit hundreds of log lines behind the exfiltration that ends it, yet that is exactly the link the model must retain.

The vanishing signal

across long gaps the learning signal shrinks toward zero, so a plain RNN cannot tie distant events together.

The LSTM adds gated memory

The remedy adds a protected memory lane with small gates that decide what to retain, what to forget, and what to pass on. Important information can persist untouched across many steps.

forget gate
Discard the stale
releases what no longer matters.
input gate
Store the new
writes in what is worth remembering.
output gate
Read what is needed
exposes only the part relevant now.
from the paper

Hochreiter & Schmidhuber (1997)[2]: "LSTM can learn to bridge minimal time lags in excess of 1000 discrete-time steps" by enforcing "constant error flow through constant error carousels," directly solving the vanishing signal.

In code these gates are provided already: an LSTM layer in TensorFlow / Keras[4] is a single line, shown next.

The cell state is a protected lane

The LSTM's real trick is a second track, the cell state, that runs straight down the sequence. The gates only make small edits to it, so important information rides it untouched across many steps.

This is the standard LSTM cell. The cell state runs straight across the top, and the gates make only small edits to it: forget removes the stale (a × on the lane), input writes the new (a + on the lane), output reads out what matters now. Each σ is a gate that opens or closes, tanh shapes a value. Because the lane flows through almost untouched, the signal no longer fades, which is how the LSTM defeats the vanishing memory.

The session classifier in code

The same idea in Keras. Each log line becomes a vector, an LSTM reads the log in order, and one output neuron returns the verdict.

model = keras.Sequential([
    layers.Embedding(vocab, 64),          # each log line -> a vector
    layers.LSTM(64),                    # read the log in order
    layers.Dense(1, activation="sigmoid")  # intrusion? 0..1
])
model.compile(optimizer="adam",
    loss="binary_crossentropy", metrics=["AUC"])
model.fit(sessions, labels, epochs=5)
Embedding

each log line becomes a learned vector.

LSTM

reads step by step, carrying the gated memory.

Dense + sigmoid

the whole session becomes one verdict.

Many steps in, one label out: this is a many-to-one model, reading the entire session before it emits a single decision. Built with TensorFlow and Keras[4], the same toolkit used in production.

Part 03 of 03

Recurrence in practice

Variants, shapes, and where these models run for a defender.

One in, one out, or many

The same recurrent core does different jobs by changing only what you read off it: a single verdict, a label at every step, or a whole new sequence.

many to one
▪ ▪ ▪ ▪ → ●
read the whole sequence, emit one verdict.
e.g. flag a session as intrusion, an email as spam. (the shape behind sentiment analysis.)
many to many
▪ ▪ ▪ ▪ → ● ● ● ●
a label at every step, in step with the input.
e.g. tag each log line or packet normal-or-suspicious, live. (the shape behind word-by-word tagging.)
sequence to sequence
▪ ▪ ▪ → ⟐ → ● ● ●
one sequence in, a different one out. An encoder reads, a decoder writes.
e.g. summarize a noisy alert stream into an incident report. (the shape behind machine translation.)

The engine is unchanged; only the input and output shape change to fit the task. A fourth shape, one-to-many, turns one seed into a whole sequence, like a malware DGA spinning out domains, though it is rarer on defense.

Two variants you will meet

Beyond the plain LSTM, two variants show up constantly in practice. Each is a single layer swap in code.

GRU
A lighter gated cell

two gates instead of the LSTM's three, so fewer parameters and faster training, often with matching accuracy. A common default when data or compute is tight.

Bidirectional
Read both directions

run one pass forward and one backward, then combine, so each step sees the whole session. Useful for a finished log, not for a live stream still arriving.

In Keras these are a one-word change: swap LSTM for GRU, or wrap it as Bidirectional(LSTM(...)). The reading-with-memory idea is identical; only the cell or the direction changes.

Why this matters to a defender

Reading behavior as an ordered progression, rather than a bag of events, changes what a detector can catch.

Behavior, not keywords
flags a malicious sequence even when each log line appears innocuous alone.
Long-range memory
the LSTM ties an early login to a late exfiltration across the intervening noise.
Learn normal, flag the odd
trained on ordinary sessions, it surfaces the ones that break the pattern.
security

Because it reads the session as it unfolds, an LSTM can raise the alarm mid-attack, on the order of events so far, rather than only after the damage is already done.

Where sequence models run in security

The same read-it-in-order machinery recurs across defensive tooling, wherever the signal lives in a stream rather than in any single event.

Log anomaly detection
an LSTM learns the ordinary rhythm of system logs and flags the lines that break the pattern.
Malicious domains
a character-level LSTM reads a domain letter by letter, telling generated names from real ones.
Host intrusion detection
a process's system-call sequence is a fingerprint of normal behavior; abuse breaks it.
Account takeover
a user's stream of logins and actions has a rhythm; a sudden break flags a hijacked account.
Payment fraud
a card's transactions form a sequence, and fraud shows up as a departure from its pattern.
Malware sandbox
the order of API calls a program makes separates malware families from benign software.

The same shape underlies insider-threat staging, SOC alert triage, and ransomware's burst of file operations. Log anomaly is a real system: Du et al. (2017)[3] built DeepLog, an LSTM trained on normal logs so any off-pattern entry surfaces as an anomaly.

Think it through

Design a fraud detector

You want to catch payment fraud, the tell-tale break in a card's stream of transactions, with a sequence model. Work through these four design questions.

what is one event? what is the sequence? many-to-one, or many-to-many? what is "normal," and where do labels come from?

Consider the adversary. The hardest question is usually the last: a cardholder's "normal" drifts constantly, travel, holidays, a big-ticket purchase, and confirmed-fraud labels arrive late and rare. That is the constraint every real deployment lives with.

Think it through

How would you beat it?

You are the attacker now. A defender runs an LSTM that scores each session for intrusion. Carry out the same attack without tripping it. How?

pad with benign actions go slow and low across days split the attack across accounts mimic a normal admin's routine

Consider the adversary. Every one of these dilutes or hides the ordered signal the model keys on. The next slide is the defender's side of exactly this arms race.

Capable, but not without limits

Slow to train

recurrence runs step by step, so it cannot parallelize the way later models do.

Data intensive

learning the shape of normal sessions requires many labeled or logged sequences.

Padding evasion

an attacker can bury the malicious steps among many benign log lines to dilute the signal.

Length limits

very long sessions still strain the memory, and not everything can be retained indefinitely.

Reading order is a genuine advantage, and a genuine arms race.

Think it through

Would you ship it?

Your LSTM works in the lab. In production it flags 200 sessions a day, and 95 percent are false alarms. Analysts are drowning. What do you change before this goes live?

raise the alarm threshold weigh a miss against a false alarm put a human in the loop retrain as behavior drifts

Consider the base rate. Real attacks are rare, so even a strong model floods the queue with false positives. The threshold and the review workflow matter as much as the model itself.

One model is one layer

A real defense never rests on a single model. Different architectures fit different data, and you chain them, with a human at the end, so no one blind spot sinks the system.

layer 1
Cheap filter
cull the obvious cheaply, so heavier models only see what is left.
→
layer 2
The right model per shape
a plain net for tables, a CNN for images, embeddings for text, an RNN for sequences.
→
layer 3
Human analyst
decides the ambiguous few at the top of the queue; volume to the model, judgment to the human.
security

Defense in depth. Each layer covers the others' blind spots: evade the sequence model and an attacker still faces the filter, the other models, and the analyst. Layering does not remove risk, it raises the attacker's cost.

One limit remains

Even the LSTM squeezes the whole session into a single memory. For a very long, detailed session, that one summary is still a bottleneck.

the remaining squeeze

a whole session → one fixed summary

reading a long report with a single sticky note; whatever it records, the rest is gone.

The next idea, attention, keeps every step in reach and looks back at whichever ones matter, rather than trusting one summary.

It removes this last limit and opens the road to transformers. That is where the next session begins.

Key terms

Sequence

data where order matters: a log, a session, a sentence.

Hidden state

the running memory of everything read so far.

RNN

a recurrent network: a loop that carries that memory forward.

Vanishing signal

the learning signal fading over long gaps, so early steps are lost.

LSTM

a gated memory cell that holds information across long gaps.

Cell state

the LSTM's protected memory lane, carried across many steps.

The core ideas

  1. In a log or a session, order carries the meaning.
  2. An RNN is one neuron, features times weights plus a bias, reused at every step with a memory.
  3. That memory fades; the LSTM gates hold it across long gaps on a protected cell state.
  4. Variants like GRU and bidirectional fit the task; attention is the next step.
