Sequences
& Recurrence

How recurrent networks read logs and event streams one line at a time, carry a memory, and how the LSTM keeps that memory alive across a long session.

Deep Learning for Cyber  ·  Francisco Laplace

What this session covers

  1. Why order carries the meaning in a log or a session
  2. The recurrent neuron and reading with a memory
  3. Why that memory fades and how the LSTM holds on
  4. Variants, shapes, and where recurrence runs in security

What makes data a sequence

A sequence is an ordered list of items in which position is itself information.

A shopping list is a bag

milk, eggs, bread. Shuffled, the groceries come out the same.

A recipe is a sequence

crack, whisk, then heat. Shuffled, breakfast fails.

A model that counts what appeared sees the bag; a sequence model reads the order.

So much data is already a sequence

Once order counts as information, much of security data turns out to be a sequence.

Text and email
words, one after another.
A login session
commands, in the order they run.
Network traffic
packets, in the order they cross the wire.
System logs
events, in time order.
An executable
bytes, read front to back.

An attack unfolds as an ordered sequence

A system log is a sequence in time, and the order carries the meaning.

08:01:07failed password for admin (x18)
08:14:22accepted password for admin
08:15:03sudo: admin opened root shell
08:22:41scp 2.1 GB to 185.62.40.9

Each line alone is ordinary. Admins mistype passwords, sign in, escalate, and copy files every day. Read in order, they trace a clear progression: brute force a login, break in, escalate to root, and exfiltrate the data.

A model that only counts which lines appeared would miss the progression entirely.

Reading in order

Why a sequence needs a memory rather than just a tally.

Think it through

Sequence, or not?

Order does not carry the signal everywhere. For each task, decide: does the meaning live in the order of events, or would counting features do just as well?

is this one email phishing? did this account get taken over? is this file malware? is this a fraud ring?

Where it bites. Two of these hinge on order, a login-and-action history and a chain of transactions; two are often fine as a bag of features. Reaching for an RNN when order does not carry the signal just adds cost and fragility.

Predictive text is a sequence model

A phone keyboard reads what has been typed so far, in order, and suggests the next word: a sequence model with a memory, in daily use.

airport
office
gate
See you at the |

reads the words so far, then predicts the next

The model reads in order, holds a running sense of the sentence, and guesses what follows.

Swap the sentence for a log, and predict the next word becomes predict the next event. When the real next event is not the expected one, that surprise is the defender's signal.

Reading with a memory

A sequence model reads one event at a time, keeping a running summary of everything before it.

That running summary is the hidden state, the model's memory of the sequence so far, carried forward and updated at every step.

\( \mathbf{h}_t = f(\mathbf{h}_{t-1},\, \mathbf{x}_t), \)

where \( \mathbf{h}_t \) is the updated memory after step \( t \), \( \mathbf{h}_{t-1} \) the memory from the previous step, \( \mathbf{x}_t \) the event read at that step, and \( f \) the update rule reused at every step; the counter \( t \) advances by one each step, so \( t-1 \) is one step earlier.

Part 02 of 03

Recurrent networks

A loop that carries memory forward, and the moment it forgets.

A loop that reuses one cell

Recurrent means a single small cell with a loop: its memory feeds back as an input to the next step, so the same cell, with the same weights, runs at every step.

Inside the cell is a familiar neuron

Opening up \( f \) reveals nothing new: the same neuron as every network so far, inputs times weights, plus a bias, through an activation.

The feedforward neuron
\[ h = \phi\!\left( w_1 x_1 + w_2 x_2 + \cdots + b \right), \]

where \( h \) is the neuron's output, \( x_1, x_2, \dots \) are the input features, \( w_1, w_2, \dots \) the weight on each, \( b \) the bias, and \( \phi \) the activation.

The recurrent neuron
\[ \mathbf{h}_t = \phi\!\left( \mathbf{W}_x\, \mathbf{x}_t + \textcolor{#3b56d8}{\mathbf{W}_h\, \mathbf{h}_{t-1}} + \mathbf{b} \right), \]

where \( \mathbf{h}_t \) is the memory after step \( t \), \( \mathbf{x}_t \) the event at that step, \( \mathbf{h}_{t-1} \) the memory from the previous step, \( \mathbf{W}_x \) and \( \mathbf{W}_h \) the weights on the input and on the memory, \( \mathbf{b} \) the bias, and \( \phi \) the activation, usually \( \tanh \), keeping each value between −1 and 1.

Think it through

Predict the next event

The model has read three log lines so far: a burst of failed logins, then an accepted login, then a sudo to root. Two questions: what would an ordinary administrator session do next, and which next line should the model find alarming?

open a config file restart a service log out install an update copy 2 GB to an outside host

Consider the adversary. Each option looks ordinary on its own. The model's only edge is the three lines it already read, the memory of how this session began.

What the memory actually holds

Read in plain English, the hidden state builds a story as each log line arrives.

08:01failed logins
→
someone is guessing passwords
08:14login: admin
→
and now they are in
08:15sudo: root
→
with full administrator control
08:222 GB copied out
→
and data is leaving, this is an attack

By the last line, four ordinary events have become one clear story, something a tally of events could never tell.

from the paper

Elman (1990)[1] set the template: "One approach is to represent time implicitly by its effects on processing rather than explicitly," letting a network's own memory carry the sequence.

The memory fades over long sequences

As the chain grows long, the early steps wash out. By the end of a long session, the network has all but forgotten how it began.

The effect is the game of telephone: a message whispered down a long line arrives garbled, and the first words are lost.

The successful login that opens an intrusion can sit hundreds of log lines behind the exfiltration that ends it, yet that is exactly the link the model must retain.

The vanishing signal

across long gaps the learning signal shrinks toward zero, so a plain RNN cannot tie distant events together.

Repeated multiplication erases the signal

Training unrolls the chain and passes the learning signal back through every step: backpropagation through time. Each step multiplies that signal by a similar factor, and Bengio, Simard & Frasconi (1994)[2] showed the consequence is fundamental.

The signal vanishes
\( 0.9^{30} \approx 0.04 \)

a factor just below 1, met thirty times, all but erases the signal; distant events cannot be linked.

The signal explodes
\( 1.1^{30} \approx 17 \)

a factor just above 1 compounds upward, and training destabilizes.

in practice

The exploding side has a standard fix: gradient clipping caps the signal at a maximum size. The vanishing side has no such patch; it takes a new cell design.

The LSTM adds gated memory

The remedy is a protected memory lane governed by small gates that decide what to retain, what to forget, and what to pass on.

Forget gate
releases what no longer matters.
Input gate
writes in what is worth remembering.
Output gate
exposes only the part relevant now.
from the paper

Hochreiter & Schmidhuber (1997)[3]: "LSTM can learn to bridge minimal time lags in excess of 1000 discrete-time steps" by enforcing "constant error flow through constant error carousels," directly solving the vanishing signal.

In code the gates come built in: an LSTM layer in TensorFlow / Keras[4] is a single line.

The cell state is a protected lane

The LSTM's real trick is a second track, the cell state. The gates make only small edits to it, so important information rides it untouched across many steps.

\( \mathbf{c}_t = \mathbf{f}_t \times \mathbf{c}_{t-1} + \mathbf{i}_t \times \tilde{\mathbf{c}}_t, \)

where \( \mathbf{c}_t \) and \( \mathbf{c}_{t-1} \) are the lane's new and previous memory, \( \tilde{\mathbf{c}}_t \) the candidate memory shaped by \( \tanh \), \( \mathbf{f}_t \) the forget gate (the \( \times \) on the lane, 0 to 1), and \( \mathbf{i}_t \) the input gate (feeding the \( + \), 0 to 1).

The session classifier in code

The same idea in Keras. Each log line becomes a vector, an LSTM reads the log in order, and one output neuron returns the verdict.

model = keras.Sequential([
    layers.Embedding(vocab, 64),          # each log line -> a vector
    layers.LSTM(64),                    # read the log in order
    layers.Dense(1, activation="sigmoid")  # intrusion? 0..1
])
model.compile(optimizer="adam",
    loss="binary_crossentropy", metrics=["AUC"])
model.fit(sessions, labels, epochs=5)
Embedding

each log line becomes a learned vector.

LSTM

reads step by step, carrying the gated memory.

Dense + sigmoid

the whole session becomes one verdict.

Many steps in, one label out: a many-to-one model reads the entire session before it emits a single decision.

in practice

Real sessions differ in length, so before batching they are padded or truncated to one fixed length. That choice matters: truncate too aggressively and the model never sees the events that gave the session away.

Part 03 of 03

Recurrence in practice

Variants, shapes, and where these models run for a defender.

One in, one out, or many

The same recurrent core does different jobs depending on what is read off it: a single verdict, a label at every step, or a whole new sequence.

Many to one
▪ ▪ ▪ ▪ → ●
read the whole sequence, emit one verdict.
e.g. flag a session as intrusion, an email as spam. (the shape behind sentiment analysis.)
Many to many
▪ ▪ ▪ ▪ → ● ● ● ●
a label at every step, in step with the input.
e.g. tag each log line or packet normal-or-suspicious, live. (the shape behind word-by-word tagging.)
Sequence to sequence
▪ ▪ ▪ → ⟐ → ● ● ●
one sequence in, a different one out. An encoder reads, a decoder writes.
e.g. summarize a noisy alert stream into an incident report. (the shape behind machine translation.)

Two variants in wide use

Beyond the plain LSTM, two variants appear constantly in practice; each is a single layer swap in code.

GRU

the gated recurrent unit. Two gates instead of the LSTM's three, so fewer parameters and faster training, often with matching accuracy. A common default when data or compute is tight.

Bidirectional

one pass forward and one backward, combined, so each step sees the whole session. Useful for a finished log rather than a live stream still arriving.

In Keras these are a one-word change: swap LSTM for GRU, or wrap it as Bidirectional(LSTM(...)).

Why this matters to a defender

Reading behavior as an ordered progression, rather than a bag of events, changes what a detector can catch.

Behavior over keywords
flags a malicious sequence even when each log line appears innocuous alone.
Long-range memory
the LSTM ties an early login to a late exfiltration across the intervening noise.
Learn normal, flag the odd
trained on ordinary sessions, it surfaces the ones that break the pattern.
security

Because it reads the session as it unfolds, an LSTM can raise the alarm mid-attack, on the order of events so far, rather than only after the damage is already done.

Where sequence models run in security

The same read-it-in-order machinery recurs across defensive tooling, wherever the signal lives in a stream rather than in any single event.

Log anomaly detection
an LSTM learns the ordinary rhythm of system logs and flags the lines that break the pattern.
Malicious domains
a character-level LSTM reads a domain letter by letter, telling generated names from real ones.
Host intrusion detection
a process's system-call sequence is a fingerprint of normal behavior; abuse breaks it.
Account takeover
a user's stream of logins and actions has a rhythm; a sudden break flags a hijacked account.
Payment fraud
a card's transactions form a sequence, and fraud shows up as a departure from its pattern.
Malware sandbox
the order of API calls a program makes separates malware families from benign software.

The same shape underlies insider-threat staging, SOC alert triage, and ransomware's burst of file operations. Log anomaly is a real system: Du et al. (2017)[5] built DeepLog, an LSTM trained on normal logs so any off-pattern entry surfaces as an anomaly.

Think it through

Design a fraud detector

You want to catch payment fraud, the tell-tale break in a card's stream of transactions, with a sequence model. Work through these four design questions.

what is one event? what is the sequence? many-to-one, or many-to-many? what is "normal," and where do labels come from?

Consider the adversary. The hardest question is usually the last: a cardholder's "normal" drifts constantly, travel, holidays, a big-ticket purchase, and confirmed-fraud labels arrive late and rare. That is the constraint every real deployment lives with.

Think it through

How would you beat it?

You are the attacker now. A defender runs an LSTM that scores each session for intrusion. Carry out the same attack without tripping it. How?

pad with benign actions go slow and low across days split the attack across accounts mimic a normal admin's routine

Consider the adversary. Every one of these dilutes or hides the ordered signal the model keys on. The next slide is the defender's side of exactly this arms race.

Capable, but not without limits

Slow to train

recurrence runs step by step, so it cannot parallelize the way later models do.

Data intensive

learning the shape of normal sessions requires many labeled or logged sequences.

Padding evasion

an attacker can bury the malicious steps among many benign log lines to dilute the signal.

Length limits

very long sessions still strain the memory, and not everything can be retained indefinitely.

Reading order is a genuine advantage, and a genuine arms race.

Think it through

Would you ship it?

Your LSTM works in the lab. In production it flags 200 sessions a day, and 95 percent are false alarms. Analysts are drowning. What do you change before this goes live?

raise the alarm threshold weigh a miss against a false alarm put a human in the loop retrain as behavior drifts

Consider the base rate. Real attacks are rare, so even a strong model floods the queue with false positives. The threshold and the review workflow matter as much as the model itself.

The sequence model is one layer

The sequence model is strong on ordered attacks and blind to much else, so it never runs alone. In a real pipeline it sits between a cheap first filter and a human analyst at the end.

layer 1
Cheap filter
cull the obvious traffic cheaply, so the sequence model only scores what is left.
→
layer 2
The sequence model
reads each session in order and flags the ordered attacks a bag of features would miss.
→
layer 3
Human analyst
decides the ambiguous few at the top of the queue; volume to the model, judgment to the human.
security

Layering. Other data shapes get their own model layer, a CNN for images, embeddings for text, so evading the sequence model still leaves an attacker facing the filter, the sibling models, and the analyst. Layering does not remove risk, it raises the attacker's cost.

One limit remains

Even the LSTM squeezes the whole session into a single memory. For a very long, detailed session, that one summary is still a bottleneck.

a whole session → one fixed summary

reading a long report with a single sticky note; whatever it records, the rest is gone.

The next idea, attention, keeps every step in reach and looks back at whichever ones matter.

It removes this last limit and opens the road to transformers, where the next session begins.

Key terms

Sequence

data where order matters: a log, a session, a sentence.

Hidden state

the running memory of everything read so far.

RNN

a recurrent network: a loop that carries that memory forward.

Backpropagation through time

training the unrolled chain by passing the blame back through every step.

Vanishing signal

the learning signal fading over long gaps, so early steps are lost.

LSTM

a gated memory cell that holds information across long gaps.

Cell state

the LSTM's protected memory lane, carried across many steps.

Gate

a sigmoid valve between 0 and 1, deciding how much passes through.

The core ideas

  1. In a log or a session, order carries the meaning.
  2. An RNN is one neuron, features times weights plus a bias, reused at every step with a memory.
  3. That memory fades; the LSTM gates hold it across long gaps on a protected cell state.
  4. Variants like GRU and bidirectional fit the task; attention is the next step.
