A network that trains is the easy part. A network that can be trusted on real, rare, adversarial mail is the actual work.

It trained, and still does not work

Suppose we build the network, run training, and watch accuracy climb to 99%.

Yet it misses nearly every real phishing email. The loss fell, but the model learned the wrong lesson. This session concerns the gap between a model that trains and a model that works.

Memorized the training set

excels on practice, fails on fresh mail

Learned to always answer safe

because phishing is rare, that alone scores 99%

Part 01 of 03

Generalization

Preventing the network from memorizing the training emails.

When a model memorizes

training vs validation loss
training validation

When the training loss keeps falling but the validation loss flattens above it, that gap is the network memorizing the exact training emails rather than the pattern.

The analogue is a student who memorized the answer key: flawless on the practice test, lost on the real exam.

security

An overfit filter clings to last month's phishing wording. The moment attackers rephrase, it collapses, so generalization is itself a security property.

Dropout relies on no single feature

During training, a random fraction of the neurons switches off on every pass, so the network cannot lean on any single feature. At test time every neuron returns.

keras.layers.Dense(16, activation="relu"),
keras.layers.Dropout(0.3),   # drop 30% each pass
keras.layers.Dense(8,  activation="relu"),

The analogue is a team that rehearses with random members sitting out. No one becomes irreplaceable, so the group stays robust.

from the paper

Srivastava et al. (2014)[2]: "the key idea is to randomly drop units (along with their connections) from the neural network during training," which keeps neurons from co-adapting and sharply cuts overfitting.

The code throughout this deck uses Keras, scikit-learn, and imbalanced-learn.[1]

Weight decay keeps the weights small

We add a small penalty on large weights to the loss. The network now has two objectives: fit the data and keep its weights modest. A large weight survives only if it genuinely earns its place.

total loss = error on the data + penalty for large weights
\[ L + \lambda \sum_i w_i^2, \]

where \( L \) is the error on the data, \( \sum_i w_i^2 \) the sum of the squared weights, and \( \lambda \) the strength of the penalty; smaller weights give a smoother decision surface, the kind that survives contact with new mail. This is known as L2 regularization.

from the paper

Krogh & Hertz (1992)[3] showed that a simple weight decay can improve generalization by suppressing weights that only fit noise. Decades on, it is still standard practice.

Early stopping halts at the best point

The validation loss is monitored every epoch. The moment it stops improving and begins to climb, training halts and the best weights so far are kept.

This is the cheapest regularizer available: no extra layers, no penalty term, only a decision not to train past the point where the model begins to memorize.

stop = keras.callbacks.EarlyStopping(
    monitor="val_loss",
    patience=3,                 # wait 3 epochs
    restore_best_weights=True,
)
model.fit(..., callbacks=[stop])
from the paper

Prechelt (1998)[4], "Early Stopping, But When?", makes the case that halting at the validation minimum is a simple, reliable guard against overfitting.

Part 02 of 03

Smoother training

Scaling, normalization, and optimizers that reach the minimum faster.

Placing features on the same scale

Suppose one feature runs 0 to 2 links and another runs 0 to 500 words. Left unscaled, the large numbers dominate the small ones and training slows.

The analogue is comparing prices in different currencies. Converting them all to a common currency first makes the comparison fair.

After scaling, every feature carries equal weight, and the network learns more evenly and faster.

from sklearn.preprocessing import StandardScaler

X = StandardScaler().fit_transform(X)

Standardization subtracts the mean \( \mu \) and divides by the spread \( \sigma \), centering each feature at zero:

\[ \hat{x} = \dfrac{x - \mu}{\sigma}. \]
in practice

The scaler is fit on training data only, then applied to validation and test. Fitting on everything leaks information from the exam into the study set.

Normalization inside the network

Scaling fixes the inputs once. Deeper in the network, each layer's numbers drift as training alters the layers before it. Normalization layers re-center those numbers during training.

Batch normalization
re-centers each layer's outputs across the batch. Trains faster and tolerates larger learning rates.
Layer normalization
normalizes across each single example rather than the batch. The variant used by transformers and LLMs.
from the paper

Ioffe & Szegedy (2015)[5]: they "refer to this phenomenon as internal covariate shift, and address the problem by normalizing layer inputs."

From plain steps to Adam

Gradient descent steps downhill. Three refinements make those steps smarter, and the last is almost always the default choice.

SGD
Plain steps
one fixed step size for every weight
→
+ momentum
Build speed
recent steps are remembered, rolling past small bumps
→
Adam
Per weight steps
each weight adapts its own step size
from the paper

Adam (Kingma & Ba, 2015)[6]: optimization "based on adaptive estimates of lower-order moments," the running mean and variance of the gradient. It is the sensible default; tuned SGD is reserved for cases that call for it.

Initialization and learning rate schedules

Careful initialization

Weights begin as random numbers, but the spread of those values matters. A poor choice causes signals to vanish or explode as they pass through layers. Good schemes set the spread from each layer's size so the signal stays healthy all the way through.

Learning rate schedules

Large steps early cover ground, then the step size shrinks near the minimum so the model settles rather than oscillates. A warmup followed by decay is the common shape.

from the paper

Glorot & Bengio (2010)[7] and He et al. (2015)[8] derived the initialization spreads, He's tuned for ReLU, that make very deep networks trainable at all.

Part 03 of 03

Imbalanced security data

When the target class is rare, and the costs of the two errors are uneven.

Why accuracy misleads

Suppose phishing is 1 email in 100. A model that marks every email safe is then 99% accurate and catches nothing. This is the defining shape of security data.

100 emails, one is phishing. A filter that flags none is still 99% accurate.

False negative

phishing reaches the inbox, and the user clicks the link

False positive

legitimate mail is quarantined and a real invoice is lost

The relevant measures are recall and precision, never raw accuracy.

\[ \text{recall} = \dfrac{\text{caught}}{\text{all phishing}}, \qquad \text{precision} = \dfrac{\text{genuine}}{\text{all flagged}}. \]

Directing a model toward the rare class

Three training-time levers shift the network's attention toward the rare phishing examples that raw accuracy lets it ignore.

Class weights
make each phishing error count for more in the loss. One line: class_weight={0:1, 1:20}.
Resampling
present more phishing (oversample) or less safe mail (undersample) so batches are less skewed.
Focal loss
automatically concentrates training on the hard, rare cases the model repeatedly misclassifies.
model.fit(X_train, y_train,
          class_weight={0: 1, 1: 20})   # each phishing error counts 20x
class_weight

the one-line lever: the loss treats every missed phishing email as twenty ordinary errors.

from the paper

SMOTE (Chawla et al., 2002)[9] over-samples "by creating synthetic minority class examples" between near neighbors; focal loss (Lin et al., 2017)[10] reshapes cross-entropy so it "down-weights the loss assigned to well-classified examples."

Think it through

Set the cutoff

A trained filter scores each email from 0 to 1, and a cutoff turns that score into a decision. Suppose the same model runs in two places: a bank's fraud desk, and a sales team's inbox where a quarantined customer email means lost revenue. For each, which error hurts more, and which way should the cutoff move?

what does a missed attack cost? what does a false alarm cost? raise the cutoff, or lower it?

Weigh the costs. A cutoff is chosen by whoever bears the costs, and the two deployments answer differently. Neither answer is 0.5. The next slide sweeps the dial and shades both errors.

The cutoff is a security decision

The model outputs a 0 to 1 score. The default 0.5 cutoff is a dial: lowering it catches more attacks at the cost of more false alarms, while raising it does the reverse.

For a bank's fraud filter a missed attack is catastrophic, so the cutoff is lowered and the false alarms are accepted. The right setting is a business and risk judgment rather than a default.

security

The threshold is chosen against the cost of each error rather than for maximum accuracy. It is revisited as attack rates shift; a fixed cutoff silently degrades as the base rate moves.

Key terms

Overfitting

memorizing the training data instead of the pattern.

Dropout

switching off neurons at random during training for robustness.

Weight decay

a penalty that keeps weights small and the model simple.

Batch / layer norm

re-centering a layer's numbers to keep training stable.

Class imbalance

when one label (phishing) is far rarer than the other.

Class weights

making the rare class count for more in the loss.

Focal loss

a loss that concentrates training on the hard, rare cases.

Threshold

the score cutoff for labeling phishing, a cost decision.
